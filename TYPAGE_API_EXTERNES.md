# Plan d'action — typage des données externes (`any` → types stricts)

Contexte : 334 occurrences de `any`/`Record<string, any>` réparties sur 23 fichiers dans `frontend/app/actions/`. La majorité vient du parsing brut des réponses JSON de MusicBrainz/Wikipedia/Wikidata/Last.fm sans interface définie. Pas une faille de sécurité, mais un risque de crash silencieux si une API externe change de forme, et une perte de garde-fou du compilateur.

## Règle pour chaque lot
- On ajoute/précise des types uniquement — aucune logique ne doit changer.
- TypeScript est effacé à la compilation : si ça compile et que les tests/usage manuel passent, le comportement runtime est identique.
- Après chaque lot : `npm run build` (ou `tsc --noEmit`) + test manuel rapide de la fonctionnalité concernée avant de passer au lot suivant.
- Ne pas toucher aux `catch (e: any)` ni aux `as any` qui contournent les types Supabase générés — ce sont des problèmes différents, à traiter séparément si besoin (regénérer les types Supabase).

## Lots (par ordre de priorité — sources externes d'abord)

- [x] **Lot 1 — `musicbrainz.ts`** (30 `any`) — fait. Interfaces ajoutées pour `/release-group`, `/artist`, `/recording`, `release-group` lookup, browse releases, Wikidata, `relations` MB. `tsc --noEmit` passe sans erreur. Les 3 `as any` Supabase restent (hors scope, catégorie différente).
- [x] **Lot 2 — `lastfm.ts` + `rym.ts`** — fait. `lastfm.ts` : `lastfmRequest` rendu générique + interfaces `LastfmTopAlbumsResponse`/`LastfmAlbumItem` pour la réponse `user.gettopalbums`. `rym.ts` n'avait qu'un `as any` Supabase (hors scope, rien à faire). `tsc --noEmit` passe.
- [x] **Lot 3 — `search.ts`** — fait, mais nature différente du prévu : ce fichier n'appelle aucune API externe, ce sont des requêtes Supabase internes castées `as any` car `search_vector`/RPC fuzzy_search_* ne sont pas dans les types générés (migration en attente). Les casts client restent (hors scope, comme `musicbrainz.ts`), mais les lignes retournées sont maintenant typées (`AlbumRow`, `ArtistRow`, `ProfileRow`, `TrackRow`, `Fuzzy*Row`) au lieu de `forEach((x: any) => ...)`. `tsc --noEmit` passe.
- [x] **Lot 4 — `metadata.ts` + `artists.ts`** — fait. `metadata.ts` : interfaces `WikidataSitelinksResponse` et `LastfmAlbumInfoResponse` ajoutées ; fix du cast `albumGenreRows as any[]` via type predicate ; `metaRow as any` conservé + commenté (colonnes `tags_checked_at`/`tag_attempts` pas encore dans les types générés, cf. `supabase_migration_tag_retry.sql`). `artists.ts` : tous les `as any` sur la table `artists` retirés sauf un (upsert avec `onConflict: 'mbid'` — le type généré exige `name` même sur update, limitation Supabase réelle, gardé + commenté). `tsc --noEmit` passe.
- [x] **Lot 5 — `track-diary.ts`** — fait. Comme `search.ts`, ce fichier est 100% Supabase interne, sans appel API externe. ~29 des 75 `any` sont des casts client `(supabase as any).from(...)` légitimes (tables `track_diary_*`/`track_stats` absentes des types générés) — laissés en place, hors scope. Les ~46 restants (formes de lignes/relations imbriquées tracks→albums→artists, profils, commentaires) sont typés via des interfaces partagées (`ArtistRef`, `AlbumRef`, `TrackRef`, `ProfileRef`, `CommentRow`, `TrackReviewRow`, `LikeFanoutEntryData`). Un cast `'track_diary_entry' as any` était carrément inutile (le type de `fanoutEvent` l'acceptait déjà) — supprimé. `tsc --noEmit` passe.
- [x] **Lot 6 — `diary.ts` + `lists.ts`** — fait. Contrairement au lot 5, la majorité des `any` ici étaient des casts superflus sur des requêtes déjà typées (`diary_entries`, `diary_comments`, `user_lists`, `list_items` sont dans les types générés) — supprimés directement. Seuls les casts sur `user_blocks` (diary.ts) et `saved_lists`/`track_diary_*` (diary.ts, lists.ts) restent hors scope (tables absentes des types générés). `lists.ts` : `attachListMeta`/`resolveListId` typés avec le vrai `SupabaseClient<Database>` au lieu de `any`. Quelques `string | null` vs `string` exposés par la suppression des casts ont été résolus avec des assertions non-null justifiées par les filtres `.not(col, 'is', null)` qui les précèdent. `tsc --noEmit` passe.
- [x] **Lot 7 — `feed.ts` + `explore.ts`** — fait. Même mélange que les lots précédents : casts client légitimes sur `user_blocks`/`track_diary_*`/`recommendation_feedback`/`user_track_recommendations`/`saved_lists`-like tables (hors types générés, conservés) + nombreux casts superflus sur des requêtes déjà typées (`diary_entries`, `feed_events`, `user_recommendations`...), supprimés. Exception notable : `getCommentActors` avait un vrai mismatch de type Supabase (relation `user:profiles!user_id` inférée en tableau alors qu'elle est un objet unique à l'exécution) — gardé un cast précis (pas `any`) documenté plutôt qu'une correction risquée. `mapFeedEvent(raw: any)` et le traitement des `payload` JSONB dans `getMyFeed`/`hasUnseenActivity` restent en `any` : polymorphisme réel selon le type d'événement, pas de la négligence. `tsc --noEmit` passe sur les deux fichiers.
- [x] **Lot 8 — petits fichiers restants** — fait (profile, social, importAllArtistAlbums, moderation, followers, following, tracks, saved-albums, curator, recommendations, avatarActions, externalImport). Casts client légitimes conservés (`user_blocks`, `content_reports`, `recommendations`/`notifications`, `external_imports`, `curator_picks` — hors types générés). `moderation.ts` : `.from(table as any)` remplacé par un vrai type `keyof Database['public']['Tables']`. **Bug réel trouvé** : `tracks.ts` (`getTrack`) et `saved-albums.ts` (`getUserSavedAlbums`) filtraient/affichaient `albums.type` sans jamais le sélectionner dans la requête — toujours `undefined` à l'exécution, masqué par `any` (le filtre anti-"Single" ne faisait donc rien). Corrigé en ajoutant `type` au `select()` ; ces deux requêtes gardent un cast client car `albums.type` n'est pas non plus dans les types générés. `tsc --noEmit` passe sur tout le projet.

## Hors scope pour l'instant
- `as any` sur le client Supabase (`createSupabaseAdmin() as any`, `(supabase as any)`) — signe que les types générés Supabase sont désynchronisés ou incomplets. À traiter séparément (régénérer `database.types.ts`), pas dans ce plan.
- `catch (e: any)` — légitime en JS/TS pour les erreurs, pas une priorité.
